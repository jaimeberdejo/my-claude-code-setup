# ADR-010: The installed guard-test footprint is opt-in (`--with-tests`)

Date: 2026-07-19
Decision: A default `install.sh` ships only `scripts/test-evidence.sh` (the runtime evidence producer the tick/grade path uses) and `scripts/test-hooks.sh` (the hook smoke `SCAFFOLD.md` documents). The full guard-test suite — the other ~27 `scripts/test-*.sh` — plus `scripts/run-guard-tests.sh` install only with `--with-tests`, which `--with-ci` implies (the shipped CI workflow runs the suite, so it must travel with it). `sync.sh` mirrors the gate: it updates the suite where a project already tracks it, but never re-adds it to a lean project (which would classify ~27 files as "new"). `doctor.sh` is unchanged — its required-file set is manifest-derived and already keeps only `test-evidence.sh` among the tests.
Why: the guard suite verifies the toolkit's own gates; it is maintainer/CI infrastructure, not a runtime dependency — a lean project still gets every gate (hooks, `tick.sh`, high-stakes, secret-scan) without it. Shipping ~28 test scripts into every user project was avoidable footprint with no runtime or debugging value for most consumers, while local verification stays one flag away (`--with-tests`, then `bash scripts/run-guard-tests.sh`, documented in `SCAFFOLD.md`). The rejected alternatives: shipping all tests always (keeps a footprint no consumer asked for) and tying the suite only to `--with-ci` (forces a user who wants local validation to also adopt the CI workflow). A dedicated `--with-tests` implied by `--with-ci` keeps CI and its suite consistent while letting a project opt into local verification alone.
